1. WHAT THIS PROJECT IS
================================================================================

Cornell MEM Schedule Planner is a free, browser-based web application for Cornell
University M.Eng. Management (MEM) distance-learning students. It helps students:

  • Build a personalized semester-by-semester degree plan
  • Track completed courses and remaining requirements
  • Handle Fall 2026 curriculum changes (returning vs incoming students)
  • Automatically schedule courses with prerequisite and credit-limit rules
  • Drag-and-drop courses between semesters to customize the plan
  • Export a filled-out copy of Cornell's official degree proposal Excel template

The app is NOT affiliated with or endorsed by Cornell University. It is a student-
built tool that mirrors Cornell's official proposal spreadsheet format.

Program facts baked into the app:
  • Minimum degree requirement: 30 credits
  • Typical completion: 4–6 semesters (~2 years)
  • Default credit load: 8 credits/semester (configurable 4–12)
  • Summer semesters: always capped at 2 credits
  • Minimum specialization electives: 2 courses total (completed + planned)

4. TECH STACK
================================================================================

Frontend:
  • React 19.1 + TypeScript 5.8
  • Vite 6.3 (build tool + dev server)
  • React Router DOM 7.18 (client-side routing: / and /stats)
  • ExcelJS 4.4 (read/write .xlsx in browser — export + curriculum import)
  • Plain CSS (src/App.css, src/index.css) — no UI framework

Backend (feature request email only):
  • Vercel serverless function: api/feature-request.js
  • Express 5.1 (optional self-hosted production: server/index.js)
  • Nodemailer 7.0 (Gmail SMTP) OR Resend 4.5 (Vercel-friendly email API)
  • Formidable 3.5 (multipart form parsing for file attachments)

Deployment:
  • Vercel (framework preset: Vite)
  • vercel.json SPA rewrite: all non-/api/* routes → /index.html

No database. No user accounts. No server-side session storage. All planner state
lives in React component state (lost on page refresh).

7. DEPLOYMENT TO VERCEL — STEP BY STEP
================================================================================

How the live site at cornellmemscheduleplanner.com is deployed:

1. SOURCE CONTROL
   Repository: https://github.com/Admft/CornellMengPlanner.git
   Push commits to GitHub; Vercel watches the repo for changes.

2. VERCEL PROJECT SETUP
   • Import project in Vercel dashboard
   • Framework preset: Vite (auto-detected)
   • Build command: npm run build  (runs tsc -b && vite build)
   • Output directory: dist
   • Install command: npm install

3. VERCEL.JSON ROUTING
   {
     "rewrites": [
       { "source": "/((?!api/).*)", "destination": "/index.html" }
     ]
   }
   This makes React Router work: all paths except /api/* serve index.html.
   The /api/feature-request.js file is auto-deployed as a serverless function.

4. ENVIRONMENT VARIABLES (Vercel Project → Settings → Environment Variables)
   Required for the feedback form to work:
     FEATURE_REQUEST_RECIPIENTS  — comma-separated email addresses
     Either:
       SMTP_HOST, SMTP_PORT, SMTP_SECURE, SMTP_USER, SMTP_PASS, SMTP_FROM
     Or:
       RESEND_API_KEY, RESEND_FROM

   Recipients are server-only — never bundled into the frontend JavaScript.

5. DOMAIN
   Custom domain cornellmemscheduleplanner.com configured in Vercel DNS settings.
   Canonical URLs in index.html and pageMeta.ts use www.cornellmemscheduleplanner.com.

6. WHAT VERCEL HOSTS
   • Static frontend from dist/ (HTML, JS, CSS, public assets)
   • public/Cornellproposal.xlsx served at /Cornellproposal.xlsx
   • Serverless function at /api/feature-request (POST, multipart form)

7. VERCEL LIMITATIONS
   • Request body limit ~4.5 MB — attachments capped at 4 MB total
   • Serverless function cold starts possible on first feedback submission
   • No persistent server — stats counters use external countapi service

8. DEPLOY TRIGGER
   Every push to the connected branch triggers automatic rebuild and deploy.
   No manual deploy steps beyond git push (assuming Vercel integration is active).

10. PLAN ENGINE — SCHEDULING LOGIC
================================================================================

Core files: src/lib/planEngine.ts, src/lib/planLayout.ts

generatePlan(state) algorithm:

1. Build course queue (buildQueue):
   a. Non-elective requirements (nonElectiveQueue):
      - All incomplete required courses (by priority pri)
      - Capstone EN5910 (pri 3) after prereqs met
      - Residential choice: EN6002 OR EN6010+EN6011 based on resChoice
      - Chosen org behavior course if not taken

   b. Electives (optimizeElectives):
      - From user's elChoices selections
      - Must pick at least (2 - already_taken_electives) courses
      - Bitmask search for subset that reaches ≥30 total credits with fewest extras
      - Prefers fewer credits when multiple subsets work
      - getSkippedElectives() returns electives left out by optimization

2. Schedule semester by semester (planFromSem → gradSem):
   For each semester:
     - Filter queue: course offered that season + prereqs satisfied
     - Greedy place courses in priority order until credit cap hit
     - Fall/Spring cap = crLimit (4–12); Summer cap = 2
     - Placed courses removed from queue; added to "done" set for prereq checks

3. Return GeneratedPlan: { plan, unscheduled, sems }

Prerequisite logic (isCourseCompleted in courses.ts):
  • EN5930 satisfied by EN5930_legacy
  • EN5941+EN5942 satisfied by EN5940
  • EN5405 waived for returning students
  • Capstone prereqs: EN5900, EN5930, EN5941, EN5942, EN5980

Drag-and-drop override (planLayout.ts):
  • planToLayout() / layoutToPlan() convert between GeneratedPlan and manual layout
  • canPlaceCourse() validates season, prereqs, credit limits
  • moveCourseInLayout() updates layout on successful drop
  • Manual layout preserved until "Recalculate" or changing grad/credits/taken

13. EXCEL EXPORT — COMPLETE TECHNICAL DESCRIPTION
================================================================================

This is one of the app's core features. Students download a filled copy of
Cornell's official MEM degree proposal spreadsheet ready for advisor review.

Template file:
  public/Cornellproposal.xlsx (~18 KB)
  Served at: https://www.cornellmemscheduleplanner.com/Cornellproposal.xlsx
  If user imported a custom .xlsx in Step 1, that file is used instead
  (cached in proposalTemplateRef in Planner.tsx)

Implementation: src/lib/excelExport.ts (uses ExcelJS in browser)
Supporting: src/lib/customExcel.ts, src/lib/planEngine.ts getAllPlacements()

Trigger:
  Step 4 → "↓ Export Excel" button → handleExport() in Planner.tsx
  → exportProposalExcel(planner, templateBuffer, displayPlan)
  → recordExcelExport() increments analytics counter

Output filename:
  {Name}_Cornell_MEM_Proposal.xlsx
  Spaces in name replaced with underscores; fallback "MEM_Proposal"

--- HEADER CELLS FILLED ---

  C4 = student name
  C5 = student ID number
  C6 = NetID
  G4 = program start semester (programStartSem, or planFromSem if blank)
  G5 = graduation semester (gradSem)
  G6 = advisor name

--- FORMULA PATCHES ---

Cornell's template has SUM formulas that incorrectly span all columns.
patchTemplateFormulas() fixes per-semester totals before writing data:

  For each column E–P:
    {col}25 = SUM({col}13:{col}24)   — required courses subtotal
    {col}33 = SUM({col}29:{col}32)   — org behavior subtotal
    {col}49 = SUM({col}36:{col}48)   — electives subtotal

  For columns H, I, J:
    {col}52 = SUM(E50:P50)           — grand total row fix

--- COURSE CREDIT PLACEMENT ---

1. Clear all course rows × all semester columns (E–P) to null

2. getAllPlacements(state, displayPlan) computes every course's semester index:
   a. Completed courses (from state.taken):
      - If takenSemesters[id] set → use that semester's column
      - Else auto-place in programStart..planFrom range respecting:
        season, prereqs, credit caps per semester
   b. Planned courses from generated/manual plan → column for that semester

3. For each placement:
   sheet.getCell(`${COL}${excelRow}`).value = course.credits
   Where COL = SEM_COLS[semIndex] (E=0, F=1, ... P=11)
   And excelRow comes from course.excelRow in catalog

--- CUSTOM COURSES (customExcel.ts) ---

For customTaken entries and courses needing non-standard rows:

Category → template row candidates:
  cap (capstone):     row 15
  org:                rows 29–32
  el (elective):      rows 36–47
  res:                rows 21–24
  req (default):      rows 13,14,16–20

If category section is full, "borrow" rows from waived/skipped courses:
  e.g. EN5405 row if returning student waived
  EN5941/5942 rows if economics requirement met via EN5940
  EN6010/EN6011 rows if resChoice is session2

For each custom course, also writes:
  B{row} = course code
  C{row} = course name
  D{row} = credits
  {col}{row} = credits (in semester column)

--- DOWNLOAD MECHANISM ---

  workbook.xlsx.writeBuffer() → Blob → temporary object URL → <a download> click
  All processing happens client-side; no server involved in export.

Example output in repo: public/Adam_Moffat_Cornell_MEM_Proposal.xlsx (sample export)

  413 — attachments too large
  503 — SMTP auth failure (with Gmail App Password hints)

Email setup:
  npm run setup:email (interactive)
  Or manual .env per .env.example

SECURITY:
  FEATURE_REQUEST_RECIPIENTS never exposed to frontend bundle.
  Only server-side env vars and serverless function can read them.

17. CURRICULUM RULES & EDGE CASES
================================================================================

RETURNING STUDENT (matriculated before Summer 2026):
  • ENMGT 5405 Applied AI waived (incoming students must take it)
  • Checkbox on Step 1; affects scheduling, Excel row borrowing, Step 2 course list

LEGACY ECONOMICS:
  • EN5940 (4 cr, old) satisfies EN5941 + EN5942 (new 1.5 cr each)
  • Checking EN5940 unchecks 5941/5942; mutual exclusion enforced

LEGACY ANALYTICS:
  • EN5930_legacy (4 cr old) satisfies EN5930 (3 cr new)
  • Mutual exclusion enforced

CREDIT SHORTFALL:
  • New 5930 (3cr) + 5941/5942 (3cr total) = 1+1 cr less than old 4+4 path
  • curriculumCreditShortfall() returns 1 or 2
  • UI suggests 1-cr or 1.5-cr electives to makeup

RESIDENTIAL REQUIREMENT:
  • EN6001 (Session I) always required
  • Second credit: EN6002 (Session II) OR EN6010+EN6011 (workshops)
  • EN6012 (legacy) + EN6011 counts as workshops done

ELECTIVE OPTIMIZATION:
  • If user selects more electives than needed, optimizer picks minimum subset
    that still reaches 30 total credits
  • Skipped electives shown in alert on Step 4

INCOMING STUDENT CREDIT OVERSHOOT:
  • Core curriculum often 32+ credits before electives (includes EN5405)
  • UI explains this is normal for non-returning students

ORG BEHAVIOR / ELECTIVE OVERLAP:
  • EN5300 and EN6030 appear in both OB and elective lists
  • If chosen as OB, corresponding elective variant (EN5300el, EN6030el) hidden
